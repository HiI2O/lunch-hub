# 実装計画: $ARGUMENTS

## 手順

1. **要件の再確認** — 何を構築するか明確化
2. **既存コードの調査** — 影響範囲と既存パターンを把握
3. **設計ドキュメント確認** — `docs/02-design/modules/` の関連ドキュメントを参照
4. **リスクの特定** — 潜在的な問題とブロッカーを表面化
5. **フェーズ分解** — 実装をステップに分割
6. **計画を提示** — ユーザーの承認を待つ

## 出力形式

```
## 要件の再確認
[箇条書き]

## 影響範囲
[変更が必要なファイル・モジュール]

## 実装フェーズ
### フェーズ1: [名前]
- ステップ...

## リスク
- HIGH/MEDIUM/LOW: [内容]

## 推定複雑度: HIGH/MEDIUM/LOW
```

NEVER 計画が承認されるまでコードを書いてはいけない。
